# Local Server (Gemma 4) 기반 Embedded Design-to-Code 풀-패키지 발표자료

> PPT 슬라이드 구성용 원고. 각 섹션 = 1~2장 슬라이드 기준.

## 1. 과제 개요

### 1-0. 프롤로그 (Storytelling)

> "새로운 로봇 개발 프로젝트가 시작되었습니다. 하지만..."

![하드웨어와 소프트웨어 통합의 어려움을 겪는 엔지니어](./robot_project_challenge_v4.png)

* **새로운 부품과 낯선 회로**: 한 번도 써보지 않은 모터 제어 전용 칩(STM32G4)과 다축 모터 구동 회로를 설계해야 합니다.
* **쏟아지는 설계 결함**: 매뉴얼 수천 페이지를 다 읽지 못해 초기 핀 매핑부터 타이머 충돌, 데드타임 오류 등 치명적인 결함이 발생합니다.
* **부족한 SW 전문가**: 어찌저찌 회로를 만들어도, 복잡한 CubeMX 설정부터 어려운 모터 제어 알고리즘(FOC, PID) 코딩까지 직접 다뤄야 하는데 사내에는 임베디드 SW 전문가가 턱없이 부족합니다.

결국 잦은 회로(PCB) 수정과 펌웨어 디버깅으로 **프로젝트 리드타임은 기약 없이 길어지고 개발자들은 지쳐갑니다.** 
"이 모든 설계-개발 과정을 AI가 전문가처럼 검증해주고 기본 코드까지 짜준다면 어떨까요?"

### 1-1. 과제 정의 (서비스명 및 주요 기능)

**서비스명**: MotorDriveForge — STM32G4 전용 Design-to-Code 자동화 Agent

**핵심 기능 (End-to-End 5-Step)**:

| Step | 기능 | 설명 |
|------|------|------|
| Step 1 | Prompt-Driven HW Verification | 사용자 의도(Prompt) + 핀맵 CSV → RAG/LLM 기반 3계층 HW 설계 검증 |
| Step 2 | Interactive Design Review | AI 분석 결과에 대한 사용자 Q&A 및 설계 확정 피드백 루프 |
| Step 3 | Standard Scaffolding | 확정된 설계 기반 프로젝트 구조 및 HAL 초기화 코드 자동 생성 |
| Step 4 | AI-Driven Driver Implementation | Gemma 4 연동, 오픈소스 Golden Module RAG 기반 드라이버 자동 구현 |
| Step 5 | Automated Build & Verify | 전체 소스 코드의 컴파일 정합성 및 동작 검증 자동화 |

**한 줄 요약**: 회로도 입력 → AI 설계 검증 → CubeMX 코드 생성 → 드라이버 자동 구현 → 빌드 검증까지 원스톱 자동화

**에이전트별 구현 및 동작 정의**:

| 구분 | [Step 1] HW 설계 검증 Agent | [Step 3-5] Code Generation Agent |
| :--- | :--- | :--- |
| **핵심 모델** | Gemma 4 31B (Dense) | Gemma 4 26B (MoE) |
| **학습/RAG 데이터** | ST 공식 문서(PDF), CubeMX XML(Pin AF), 오픈소스 회로도(KiCad) | Golden Module(C/H), HAL 예제 코드, 오픈소스 모터 프로젝트(SimpleFOC 등) |
| **구현 방법** | 규칙 엔진 + RAG 기반 하이브리드 검증 | Golden Module RAG + USER CODE 영역 스니펫 주입 및 빌드 자동화 |
| **추론/동작 기능** | 사용자 설계 의도(Prompt)와 회로 데이터(CSV)를 결합 분석하여 3계층 설계 결함 탐지 및 피드백 | 프로젝트 구조 생성, 드라이버 코드 구현, 빌드 및 검증까지의 전 과정 수행 (Step 3, 4, 5 통합) |

### 1-2. 과제 추진 배경 (AS-IS 현황 및 문제 정의)

**기존 업무 프로세스 (AS-IS)**:

```
[회로 설계] → (수동 검토) → [CubeMX 수동 설정] → (수동 코딩) → [드라이버 구현] → (수동 테스트) → [빌드/검증]
   ↑ 정보 단절        ↑ 시행착오            ↑ 반복 작업           ↑ 품질 편차
```

**문제점 3가지**:

1. **설계 오류의 늦은 발견**: 회로도의 핀 매핑 오류(예: 상보 PWM 쌍 불일치, BOOT0 플로팅)를 코드 구현 단계에서야 발견 → 설계 수정·PCB 재제작 비용 발생
2. **드라이버 구현의 높은 진입 장벽**: CubeMX가 생성하는 HAL 초기화 코드 위에 실제 모터 제어 드라이버(FOC, PID, 통신 프로토콜)를 처음부터 작성해야 함 → 주니어 개발자 시행착오 多, 프로젝트 일정 지연
3. **AI 활용 시 보안 우려**: ChatGPT 등 외부 AI 사용 시 핵심 회로 자산·제어 알고리즘 유출 위험 → 사내 폐쇄망 AI 필요

---

### 1-3. 과제 추진 목적 (TO-BE)

**TO-BE 업무 프로세스**:

```
[회로 설계] → [AI 자동 검증 + 피드백] → [.ioc 자동 생성] → [AI 드라이버 자동 구현] → [자동 빌드/검증]
               MotorDriveForge (End-to-End 자동화)
```

**목적**:
- 회로도 분석부터 드라이버 구현까지를 **하나의 유기적 파이프라인**으로 통합
- NVIDIA DGX Spark 및 Ollama 기반 **폐쇄망 내 완전 로컬 운영 (Gemma 4)**
- 초급 개발자도 **전문가 수준의 결과물**을 낼 수 있는 환경 구축

### 1-5. 기대 효과

**정량적 기대 효과**:

| 항목 | AS-IS | TO-BE | 개선율 |
|------|-------|-------|--------|
| 회로 검증 소요시간 | 2~3일 (수동 체크리스트) | 수 분 (AI 자동 검증) | **~95% 단축** |
| CubeMX 설정 + 드라이버 구현 | 2~4주 | 2~3일 | **~85% 단축** |
| 설계 오류로 인한 PCB 재제작 | 연 3~5회 | 연 0~1회 | **비용 70%↓** |
| 신입 개발자 온보딩 기간 | 3~6개월 | 1~2개월 | **60% 단축** |

**정성적 기대 효과**:

- **품질 표준화**: 사내 Best Practice(Golden Module)가 자동 반영된 코드 → 개발자별 품질 편차 해소
- **지식 자산화**: 숙련 엔지니어의 설계 노하우가 RAG 지식베이스와 검증 규칙으로 영구 보존
- **보안 확보**: 모든 처리가 사내 폐쇄망(DGX Spark) 내에서 완결 → 외부 유출 원천 차단
- **개발 문화 혁신**: AI 기반 설계 리뷰 문화 정착 → 코드 리뷰 부담 감소, 엔지니어 창의적 업무 집중

### 1-6. 확산 가능성

**단계별 확산 로드맵**:

| 단계 | 범위 | 대상 | 가치 |
|------|------|------|------|
| Phase 1 (현재) | STM32G4 모터 제어 보드 | 로봇사업부 FW팀 | 모터 드라이버 자동화 |
| Phase 2 | STM32 全 시리즈 (F4/H7/L4 등) | 전사 임베디드 개발팀 | 범용 펌웨어 자동화 |

**특히 Phase 3 — 회로 설계까지 확산**:
- 현재 "회로도 → 코드" 방향을 **"요건 정의 → 회로도 → 코드"** 로 확장
- 자연어로 "BLDC 2축, FDCAN, 엔코더"를 입력하면 → AI가 **회로도(KiCad Schematic)까지 자동 생성**
- 축적된 검증 규칙과 Golden Module을 역방향으로 활용하여 설계 제안

### 2-1. 활용 데이터

| 데이터 종류 | 원천 | 유형 | 형태/구조 | 수량 | 업데이트 주기 |

### 2-2. 진행 과정

#### 데이터 수집 및 전처리

```
[ST 공식 PDF 14건] → pdfplumber 텍스트 추출 → 섹션별 청킹 (512토큰, 64 overlap)
[CubeMX XML]      → Python XML 파싱 → pin_af_db.json (규칙 엔진 직접 조회)
[오픈소스 코드]    → 함수 단위 청킹 (정규식 경계 탐지) → 메타데이터 태깅
```

#### RAG 파이프라인 구축

```
[청킹된 문서] → BGE-M3 임베딩 (다국어) → Qdrant Vector DB (Dense 검색)
                                        → BM25 역인덱스 (Sparse 검색, 핀번호/함수명 정확 매칭)
                                        → Cross-encoder Re-ranking (상위 20 → 5개)
```

#### 시스템 프롬프트 및 모델 입출력

**모델 입력**:
- 사용자 자연어 프롬프트: "BLDC 1축 FOC, 엔코더, FDCAN 1Mbps, 내부 OPAMP"
- 핀맵 CSV: `PA8,TIM1_CH1,U_PWM_H` 형식
- RAG 검색 결과: 관련 ST 공식 규칙, 레퍼런스 회로도

**기대 출력**:
- 3계층 검증 JSON 리포트 (errors/warnings/suggestions)
- 확정 핀 JSON (Step 2 입력)
- CubeMX .ioc 파일 + HAL 초기화 코드
- Golden Module이 통합된 완성 펌웨어 .zip

#### 3-Step 파이프라인 아키텍처

```
[입력] 핀맵 CSV + 자연어 프롬프트
         ↓
[STEP 1] HW 설계 검증 Agent      ← Gemma 4 31B Dense + 3계층 규칙엔진 + RAG
         3계층: 인프라 → 모터 논리 → 페리페럴 제약
         errors > 0 → HTTP 403 차단
         ↓ PASS only
[STEP 2] CubeMX 자동화            ← 4단계 워크플로우 (LLM 최소 개입)
         .ioc 수정 → CubeMX CLI → 스니펫 주입 → ZIP
         ↓
[STEP 3-5] Code Generation Agent      ← Gemma 4 26B MoE + Golden Module RAG
         USER CODE BEGIN/END 영역에 드라이버 코드 삽입
         ↓
[출력] 완성 펌웨어 .zip
```

#### 코드 레이어 구조 (3계층 분리)

```
┌──────────────────────────────────┐  ← Step 3 LLM이 생성 (범위 최소화)
│  Layer 3. 통합 레이어             │     함수 호출, 연결 코드, 인터럽트 핸들러
├──────────────────────────────────┤  ← 오픈소스 기반 Golden Module (검증된 알고리즘)
│  Layer 2. 드라이버 알고리즘       │     FOC, PID, Clarke/Park, SVPWM, FDCAN
├──────────────────────────────────┤  ← Step 2 CubeMX CLI 자동 생성 (정확도 100%)
│  Layer 1. HAL 초기화             │     MX_TIM1_Init, MX_ADC_Init, MX_FDCAN_Init
└──────────────────────────────────┘
```

> **핵심 설계 원칙**: LLM은 HAL API를 직접 생성하지 않음. CubeMX가 보장하는 정확한 초기화 코드 위에 LLM이 알고리즘 레이어만 조립.

### 2-3. 본인의 역할

**전체 과제원 역할**:

| 구분 | 인원 | 역할 |
|------|------|------|
| 본인 (PM) | 1명 | 전체 아키텍처 설계, Local LLM 연동, 드라이버 생성 로직, 성능 검증 (기여도 70%) |
| 팀원 A | 1명 | RAG 파이프라인 구축, 데이터 수집/전처리, 벡터 DB 운영 |
| 팀원 B | 1명 | 웹 UI 개발, CubeMX CLI 연동, 빌드 자동화 |

**본인 세부 R&R**:

1. **아키텍처 설계**: 회로 분석 → Local LLM (Gemma 4) 기반 드라이버 자동 구현 간 연동 파이프라인 전체 설계
2. **프롬프트 엔지니어링**: Gemma 4 모델 활용 고난도 임베디드 드라이버 생성 프롬프트 및 Few-shot 학습 데이터 구축 주도
3. **Golden Module 설계**: 오픈소스 FOC 코드(Arduino-FOC, MESC, VESC, moteus, ODrive)를 분석하여 STM32G4 HAL 기반 표준 모듈로 가공·등록
4. **성능 검증**: 20자유도 로봇 핸드 등 다자유도 시스템의 복잡한 HW 설정 및 제어 코드 생성 성능 검증

**문제 해결 사례**:
- 기존 LLM 코드 생성의 HAL API 오류 문제 → "CubeMX CLI + LLM 접착 코드" 3계층 분리 아키텍처로 해결
- 핀 AF 할루시네이션 문제 → CubeMX XML 파싱 기반 규칙 엔진 + RAG 하이브리드 검증으로 정확도 확보

## 3. 향후 발전 방향

### 3-1. 추가 개선 사항

#### 데이터 측면

| 개선 항목 | 현재 | 목표 | 기대 효과 |
|-----------|------|------|-----------|
| 학습 데이터 규모 | RAG 기반 운영 | Fine-tuning 데이터 1,000쌍 구축 | 검증 정확도 95%↑, JSON 포맷 일관성 |
| 칩 커버리지 | STM32G4 단일 | STM32 全 시리즈 (F4/H7/L4 등) | 사용자 대상 10배 확대 |
| 사내 코드 자산 | 오픈소스 Golden Module | 사내 최적화 FOC/PID 코드로 교체 | 사내 표준 준수, IP 보호 |
| Vision 회로도 분석 | CSV 핀맵 수동 입력 | 회로도 이미지(PDF/KiCad) 직접 분석 | 입력 단계 자동화 |

#### 학습/모델 측면

| 개선 항목 | 현재 | 목표 | 기대 효과 |
|-----------|------|------|-----------|
| Fine-tuning | 미적용 (RAG only) | QLoRA Fine-tuning (Gemma 4 31B/26B) | 도메인 특화 응답 품질 향상 |
| 에러 사례 DB | 초기 구축 중 | 오픈소스 및 사내 실제 설계 오류 사례 500건+ 축적 | 실무 에러 패턴 커버리지 확대 |
| 피드백 루프 | 없음 | 사용자 수정 이력 → Few-shot 자동 전환 | 사용할수록 정확도 향상 |

#### 서비스 측면

| 개선 항목 | 현재 | 목표 | 기대 효과 |
|-----------|------|------|-----------|
| UI | Streamlit MVP | React 18 + TypeScript 프로덕션 UI | UX 개선, 실시간 WebSocket |
| 인증 | 없음 | 사내 LDAP/AD SSO 연동 | 보안 강화, 사용자별 이력 관리 |
| 멀티모터 | 1축 검증 | 2~4축 동시 검증 + 리소스 충돌 감지 | 로봇 핸드 등 다축 시스템 지원 |
| 역방향 설계 | 회로도→코드 | **요건→회로도→코드** (회로 자동 설계) | 설계 공정 전체 자동화 |

| 배포 | Docker Compose | Kubernetes + CI/CD 파이프라인 | 확장성, 가용성 향상 |
